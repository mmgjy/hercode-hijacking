# Evaluation

The jury will review both your demo and your forked repository.

## Rubric

| Area | What We Look For |
| --- | --- |
| Signal detection | The system finds plausible emerging opportunities from messy, real-world sources. |
| Evidence quality | Claims are backed by source names, URLs, artifacts, or inspectable data. |
| Transferability | The system explains whether the opportunity can work in Switzerland or DACH. |
| Business actionability | The output leads to clear retail actions, not just a trend list. |
| Reusability | Inputs, source choices, and scoring logic are configurable enough to reuse elsewhere. |
| Technical architecture | The solution is understandable, practical to run, and robust for a hackathon demo. |
| Communication | The dashboard, report, visualization, or video makes the result easy to judge. |

## Strong Submission Signals

- A clear data flow from source collection to ranked recommendations.
- Multiple evidence sources combined and deduplicated.
- Transparent confidence and limitation notes.
- A dashboard or visual explanation that makes the recommendation easy to inspect.
- A completed `SUBMISSION.md` with setup instructions.
- Code that can be reviewed from the submitted fork.

## Common Pitfalls

- Trend claims without evidence links.
- A one-off report with no reusable method.
- No explanation of why the trend transfers to Switzerland or DACH.
- No runnable instructions.
- Secrets or private credentials committed to the repository.
